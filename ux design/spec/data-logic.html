Part 03 · Data & Logic

The model behind the pixels

The UI is a faithful renderer of one data structure and a handful of deterministic rules. This part defines the ticker schema, the A/B/C indicator framework, the rule-based tier gates, the 9-state lane contract, the two-pass scan pipeline, and how state persists. Get this right and the screens fall out of it.

The mental model

Flow → indicators → tier

For each ticker, each cycle: ingest prints across lanes → compute three independent indicators → derive a signed direction from order flow → apply explicit tier gates → attach corroboration → render. No step is a black box; every number on screen traces to a field.

The A/B/C framework

Three lenses, never averaged

The product's defining decision: it shows three orthogonal readings of "how unusual" and never collapses them into one score. Each answers a different question.

APercentile

"How does this name rank against its universe right now?" 0–1 percentile on volume / focus / pressure. Requires a peer group — absent in single-ticker mode (shows N/A).

BZ-score

"How unusual is this vs the name's own history?" σ above its 20-session median on volume / notional / focus / pressure / pre-market. The primary tier gate.

CRaw metric

"What is the plain magnitude?" Unitless ratios: notional ×, volume ×, trade-count ×, focus share, largest-print multiple, net pressure %.

Direction & signing

From order flow, never price

Direction is the sign of net pressure derived by classifying each print as buyer- or seller-initiated (Lee-Ready: quote rule, then tick test; midpoint & unclassifiable prints are excluded from signing). A signing-confidence score reflects how much of the flow was confidently signed.

DirectionTagRule
Buyer-side↑bullnet signed pressure ≥ +0.60
Seller-side↓bearnet signed pressure ≤ −0.60
Mixed↔mixedboth sides present, neither dominant
Undetermined–undet|pressure| below threshold — elevated but non-directional
!
Below 50% signing confidence, the UI explicitly flags direction as "indicative only". Never present a low-confidence direction as conviction.

Tier classification

Explicit gates — the heart of the system

Tier is assigned by rules, evaluated top-down (try A, then B, then C, else D). In single-ticker mode the A (percentile) criteria are skipped — only B + C apply. These exact rules drive the "Explain this tier" modal.

AActionableclears every gate with independent corroboration
  • ✓ B ≥ 2.5σ on ≥ 2 components (volume / focus / notional)
  • ✓ A ≥ 85th percentile on ≥ 2 components (portfolio / scan only)
  • ✓ Direction consistent across components (net & block pressure same side)
  • ✓ ≥ 1 independent (Strong) corroboration (price / provider / options)
  • ✓ Lane state = ready (all required lanes fresh)
BReviewmeets threshold, wants confirmation
  • ✓ B ≥ 1.5σ on ≥ 1 component
  • ✓ A ≥ 70th percentile on ≥ 1 component (portfolio / scan only)
  • ✓ Direction present (a confirmed side)
  • ✓ Lane state = ready or partial
CContext

Activity detected (elevated volume / B), but the Tier-B threshold is not met — or the tier is capped at C because lane coverage is below 90%, or direction is unconfirmed.

DSuppressed

No tier emitted. A required lane is loading, or the sample is below ~50 prints after exclusions, or indicators can't be computed. The system refuses to guess.

Elevation rule

◆
A confirmed, direction-aligned provider alert is the only thing that elevates. A Tier-A name with a matching provider alert is "eligible for elevation"; a Tier-B name "could elevate to A once B ≥ 2.5σ and lane = ready". Contextual flags (news, macro) never elevate — they inform only.

The lane state contract

Nine states — never "stale"

Every data source is a "lane" reporting exactly one of nine states. The contract is deliberate: a lane is never vaguely "stale" — it is precisely loading, needs-refresh, partial, unavailable, or blocked, and the tier responds accordingly. This is what lets the product be honest about incomplete data.

The two-pass scan pipeline

Screen cheap, resolve expensive

Scanning thousands of names live is infeasible, so the pipeline splits the work. The UI makes both passes visible — the user watches the funnel narrow and rows resolve.

1

Pass 1 — Screen

Every constituent is cheaply pre-screened from daily bars (seconds). Coarse candidate set flagged.

2

Pass 2 — Resolve

The flagged shortlist gets live trade slices pulled and each row resolved to full A/B/C — highest-notional first, row by row.

Done — Results

Refinement bar + 3 views. Save the scan or watch all shown.

Scan criteria (the "Matched" pills): signed pressure on the chosen side ≥ 60% · B ≥ 1.5σ vs own history · notional ≥ 1.5× session median · required lanes ready.

Alert-rule shape

// one rule — evaluated every cycle against live A/B/C
{ id, name, enabled,
  scope: 'all' | 'portfolio' | 'watchlist',
  dir:   'any' | 'bullish' | 'bearish',
  minTier: 'any' | 'C' | 'B' | 'A',
  cB: { on, v },   // B ≥ v σ
  cA: { on, v },   // A ≥ v th percentile
  cC: { on, v },   // C notional ≥ v×
  provider }       // require a confirmed provider alert

A ticker matches when it passes all enabled conditions (tier D never matches). Matches appear in the feed tagged "rule" and as a sparkle flag on any table/card row for that symbol.
